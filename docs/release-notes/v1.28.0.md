# v1.28.0 — Org-Level Control Inheritance (Feature 044)

**Released**: 2026-03-22
**Feature**: 044-org-control-inheritance

---

## Overview

Version 1.28.0 introduces **Org-Level Control Inheritance** — a centralized mechanism for defining inheritance designations that automatically apply across all registered systems. Instead of manually configuring inheritance per system, organizations can derive defaults from their Security Capabilities Library, cascading standardized designations to every system baseline.

---

## New Features

### Org-Level Inheritance Defaults

- **Derive Org Defaults**: Scans org-wide capability-control mappings in the Security Capabilities Library and creates `OrgInheritanceDefault` records per NIST control with the correct inheritance type, provider, and source capability references.
- **Cascade Propagation**: Derived defaults automatically push `OrgDerived` designations to every system baseline. Controls without existing manual overrides receive org defaults; existing manual designations are preserved.
- **Automatic Re-derivation**: Org defaults are re-derived and cascaded automatically when:
  - Capability-control mappings are created or deleted
  - A capability's status changes (e.g., Active → Deprecated)
  - A capability is deleted from the library
- **Revert to Org Defaults**: Per-system bulk action to restore overridden controls back to their org-level designation.

### Designation Source Tracking

Every `ControlInheritance` record now tracks how it was set:

| Source | Description |
|--------|-------------|
| `OrgDerived` | Derived from org-level capability-control mappings |
| `OrgPropagation` | Cascaded to a system during org default derivation |
| `Manual` | Set directly by a user via inline editing |
| `BulkUpdate` | Applied via multi-select bulk update |
| `ProfileApply` | Applied from a CSP profile |
| `CrmImport` | Imported from a CRM spreadsheet |

### CRM Export Enhancement

The "Designation Source" column is now included in all three CRM export layouts:
- **Custom** — Control ID, Family, Inheritance Type, Provider, Customer Responsibility, Designation Source
- **FedRAMP** — FedRAMP template columns plus Designation Source
- **eMASS** — eMASS import format plus Designation Source

### Dashboard UI

- **Summary Bar**: Two new cards — "Org Defaults" and "Overrides" — appear when org defaults exist, showing the count of org-derived vs. system-overridden designations.
- **Source Filter**: New "All Sources" dropdown in the filter bar with options: All Sources, Org Defaults, System Overrides, Undesignated.
- **Source Badges**: Colored badges on each table row indicate designation source:
  - Teal = Org Default
  - Purple = CSP Profile
  - Sky = CRM Import
  - Gray = Manual / Bulk Update
- **Org Default Tooltips**: Teal checkmark for org-derived controls; amber warning for controls that override an existing org default (shows org default details on hover).
- **Coverage Banner**: Teal banner below the page header: "N of M controls have org-level defaults."
- **View Org Defaults Modal**: Full-screen modal listing all derived defaults with control ID, inheritance type, provider, source capabilities, and mapping role. Includes search and pagination.
- **Header Toolbar Reorganization**: When org defaults are active, "Apply CSP Profile" and "Import CRM" move into a "More Actions" (⋯) dropdown to reduce clutter.

## REST API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/dashboard/inheritance/org-defaults` | List org-level inheritance defaults (paginated, filterable) |
| POST | `/api/dashboard/inheritance/org-defaults/derive` | Derive org defaults from capabilities and cascade to all systems |
| POST | `/api/dashboard/systems/{systemId}/inheritance/revert-to-org-defaults` | Revert selected controls to org defaults |

### Enhanced Endpoints

- `GET /systems/{systemId}/inheritance` — New `source` query parameter for filtering by designation source. Response includes `designationSource` and `orgDefault` fields per item, plus `orgDefaultCount`, `systemOverrideCount`, and `sourceBreakdown` in the summary.
- `GET /systems/{systemId}/inheritance/{controlId}/audit` — Response includes `changeSourceLabel` with human-readable labels (e.g., "Org Default", "CSP Profile").

## Entity Model Changes

### New Entity: `OrgInheritanceDefault`

| Column | Type | Description |
|--------|------|-------------|
| `Id` | string(36) | Primary key (GUID) |

| `DerivedAt` | DateTime | Timestamp of derivation |

| Column | Type | Description |
|--------|------|-------------|
| `DesignationSource` | string(20) | OrgDerived, Manual, ProfileApply, CrmImport, BulkUpdate |
| `OrgInheritanceDefaultId` | string(36) | FK → OrgInheritanceDefault |

## Migration Notes

### Database Schema

Two new columns on `ControlInheritances`:
- `DesignationSource NVARCHAR(20) NULL`
- `OrgInheritanceDefaultId NVARCHAR(36) NULL` (FK to `OrgInheritanceDefaults`)

Existing rows should be backfilled with `DesignationSource = 'Manual'`.

The `OrgInheritanceDefaults` table is auto-created by EF Core migrations.

### Breaking Changes

None. All existing API endpoints maintain backward compatibility. The `designationSource` and `orgDefault` fields are additive in the inheritance list response.
