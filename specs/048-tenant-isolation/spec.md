# Feature Specification: Tenant- & Organization-Scoped Data Isolation

**Feature Branch**: `048-tenant-isolation`
**Created**: 2026-05-07
**Status**: Draft
**Input**: User description: "for onboarding, we need to keep all organizations separated. This will be used by mission owners in their own tenant and also be used by Cloud providers like Flankspeed and C-Army." Confirmed scope: dual-mode deployment (single-tenant + multi-tenant), three-level hierarchy (Tenant → Organization → System), defense-in-depth (EF query filters + endpoint scope checks + SQL Server RLS), CSP-Admin cross-tenant role, comprehensive audit of every DbSet / endpoint / MCP tool.

## Background

ATO Copilot today has a partial multi-tenant model: ~13 onboarding entities carry a `TenantId Guid` column, but the **other 60+ DbSets** in `AtoCopilotContext` — including `RegisteredSystem`, `ComplianceFinding`, `EvidenceArtifact`, `Deviation`, `PoamItem`, `RemediationTask`, `SspExport`, every assessment table, every Azure-discovered component, etc. — have **no tenant or organization column at all**. There are **zero `HasQueryFilter` calls** in the codebase, so any user with a valid CAC/Entra token can in principle query any data row. The CAC middleware reads the Entra `tid`/`oid` claims but does not surface them to a typed tenant context that services can consume.

This feature must close that gap so a single deployment can safely host:

1. **Mission-owner tenants** — one organization per deployment; staff sees only their tenant's data.
2. **Cloud Service Provider (CSP) tenants** like Flankspeed and C-Army — one deployment hosting *many* mission-owner tenants. Mission-owner staff sees only their own data; CSP-Admin staff can switch into any tenant for support.

## Clarifications

### Session 2026-05-07

- Q: How is the `CSP.Admin` role assigned and surfaced to the application? → A: Entra Security Group mapped to the `CSP.Admin` role via config (`Auth:RoleClaimMappings:CSP.Admin = <groupObjectId>`); requires `groupMembershipClaims` on the app registration to emit group object-IDs.
- Q: How is a new tenant created in MultiTenant mode? → A: Hybrid — CSP-Admin pre-provisioning API plus a Tenant-and-Organization Onboarding Wizard that captures tenant-level information; no silent auto-provisioning.
- Q: What is the runtime behavior of `Tenants.Status = Suspended` vs `Disabled`? → A: Three-state graduated — Active = full access; Suspended = reads succeed, writes return `423 TENANT_SUSPENDED`, CSP-Admin impersonation still allowed; Disabled = all requests return `401 TENANT_DISABLED`, data retained but excluded from queries unless CSP-Admin.
- Q: How do existing org-level control inheritance (Feature 044) and shared evidence (Feature 038) interact with tenant boundaries? → A: Default is strict per-tenant isolation; cross-tenant references are rejected. CSP-Admin MAY explicitly `Publish` an organization-level control narrative or evidence artifact to all tenants, which replicates it as a `[GlobalReference]` row in the system tenant; tenants then inherit only via the published global copy, never via cross-tenant FKs.
- Q: How is an existing single-tenant database migrated to a multi-tenant deployment? → A: Hybrid — ship both a built-in CSP-Admin admin endpoint (`POST /api/admin/migrate-to-multitenant` with preview support) for live deployments and a standalone `ato-cli tenant` migration command for offline / air-gapped DBs. Both perform the same idempotent backfill + RLS-install transaction.

### Session 2026-05-08

- Q: Should the hosting CSP itself have a first-use onboarding flow distinct from per-tenant onboarding? → A: Yes — when a deployment first boots in `MultiTenant` mode and no `CspProfile` row is `Active`, a CSP-Admin-only **CSP Onboarding Wizard** captures the CSP's own identity (legal entity, display name + branding, primary support contact, default classification floor). It runs once, is re-entrant, and gates all other tenant-scoped endpoints with `503 CSP_ONBOARDING_INCOMPLETE` until it completes. Per-tenant onboarding (FR-054) is unavailable until then. `SingleTenant` deployments skip this wizard entirely.
- Q: Does the CSP need a cross-tenant operational dashboard? → A: Yes — `GET /api/csp/dashboard/*` endpoints (CSP-Admin only) return aggregate KPIs (tenant count by status, total organizations, total systems, ATOs grouped by `Authorized/InProcess/Denied`, open findings by severity, open POA&Ms, open deviations) across **all** tenants without requiring per-tenant impersonation, and a paginated tenants list that drills into impersonation. The all-up dashboard reads through the CSP-Admin global-view query path (no impersonation cookie set) so it executes in a single round trip.
- Q: Should the CSP Onboarding Wizard accept the CSP's existing ATO artifacts (SSP PDF, FedRAMP package, eMASS export) and use them to seed inheritance content for hosted tenants? → A: Yes — the wizard adds an **ATO Documents** step where a CSP-Admin uploads one or more files (PDF, DOCX, OSCAL JSON, eMASS ZIP, FedRAMP SSP XLSX) up to 50 MB each. The system parses each file using existing parsers (PdfPig from Feature 047, OSCAL parser from Feature 022, ClosedXML, OpenXml), extracts component-like entries (system descriptions, services, infrastructure layers), and stores them as `CspInheritedComponent` rows in the system tenant marked `[GlobalReference]`. These components surface read-only to **every** hosted tenant and may be referenced (not copied) when authoring systems and inheritance defaults under any tenant.
- Q: Should the system attempt to map extracted components to NIST controls automatically, and what happens on failure? → A: Yes — for each extracted component the system invokes the existing AI capability-mapping pipeline (Feature 045 / Feature 008) to generate `CspInheritedCapability` rows, each linked to one or more NIST 800-53 controls. Capabilities the AI cannot map with sufficient confidence (configurable threshold, default 0.6), where the AI errors out, or where the source description is too sparse, MUST be persisted with `Status = NeedsReview` and a free-text `MappingFailureReason`. The wizard MUST show CSP-Admin a tally of `Mapped` / `NeedsReview` capabilities before they finalize submit, and a CSP-Admin can later resolve `NeedsReview` items via `PATCH /api/csp/inherited-components/{id}/capabilities/{capabilityId}/review`.
- Q: When a tenant consumes a `CspInheritedCapability`, how should the CSP's source documentation flow into the tenant's control evidence and narrative? → A: On first consumption (a tenant-local `OrgInheritanceDefault` or `ControlInheritanceMapping` row referencing the capability via a new `SourceCspCapabilityId Guid?` FK), the system MUST (a) persist exactly one tenant-local `EvidenceArtifact` of `Type = CspInheritedReference` per mapped control, payload `{ SourceCspComponentId, SourceCspCapabilityId, SourceFileName, SourceArtifactReference, IsImmutableSource = true }` (FR-107); (b) invoke the existing `IControlNarrativeService` (Feature 008 / Feature 024) with a single new CSP-context prompt fragment so the generated narrative explicitly states the control is inherited from `<CspProfile.DisplayName>` and cites the source document by name (FR-108); (c) when the AI service is unavailable, persist a deterministic stub narrative with `Status = NeedsReview` so the SSP draft is never blank (FR-109). All three steps reuse existing services — no new evidence path, no new narrative service, no new endpoint.
- Q: Should the US9 / US10 implementation introduce new mapping / narrative / parser / evidence / inheritance services, or reuse the existing ones from prior features? → A: Reuse-first. FR-110 mandates reuse of `ICapabilityMappingService` (Feature 045 / 008), `IControlNarrativeService` (Feature 008 / 024), document parsers (PdfPig — Feature 047, OSCAL — Feature 022, OpenXml, ClosedXML), `IEvidenceArtifactService` / `IEvidenceStorageService` (Feature 038), and `IOrgInheritanceDefaultService` / `ControlInheritanceMapping` (Features 044 / 043). Any code that supersedes a prior tenant-local equivalent MUST be removed in the same PR (FR-111); the [plan.md § Reuse-First Audit](plan.md) subsection enumerates each existing code path with file paths, the surgical extension required, and any redundant code identified for removal. A duplicate DI registration of any of those interfaces MUST fail startup.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Mission Owner Sees Only Their Own Tenant's Data (Priority: P1)

A mission owner from "Coastal Watch" logs in with their CAC. Every page (Systems, Components, Capabilities, Controls, Evidence, POA&Ms, Deviations, Onboarding) must show **only** Coastal Watch data, regardless of how many other tenants share the deployment. No row from another tenant may appear in any list, dashboard widget, search result, MCP tool response, document export, or audit query.

**Why this priority**: This is the contractual minimum for any multi-tenant deployment. Without P1, the system cannot legally be hosted on shared infrastructure for DoD mission owners.

**Independent Test**: Seed two tenants (T-Coastal, T-Eagle) each with their own systems, components, evidence, POA&Ms. Authenticate as a Coastal user and confirm via integration tests that every dashboard endpoint, every MCP tool that returns lists, every export, and every search returns zero T-Eagle rows.

**Acceptance Scenarios**:

1. **Given** two tenants each with 3 registered systems, **When** a Coastal mission owner calls `GET /api/dashboard/systems`, **Then** the response contains exactly the 3 Coastal systems (no T-Eagle systems, no error).
2. **Given** a Coastal-scoped session, **When** the user calls `GET /api/dashboard/systems/{eagleSystemId}` with an Eagle system ID, **Then** the API returns 404 (NOT 403 — pretend it doesn't exist to avoid leaking existence).
3. **Given** the Component Library page loaded as a Coastal user, **When** the page polls `/components`, **Then** only Coastal-owned components and Coastal Azure-discovered resources are returned.
4. **Given** an SSP export job, **When** rendered for Coastal Watch, **Then** the document contains zero data from any other tenant — including in the cover page, control narratives, evidence references, or POA&Ms.
5. **Given** a Coastal user invokes any of the 130+ MCP tools, **When** the tool reads the database, **Then** the EF query is scoped to Coastal regardless of whether the tool author remembered to add a filter.

### User Story 2 — CSP-Admin Switches Tenants for Support (Priority: P1)

A Flankspeed CSP-Admin engineer needs to investigate an issue in the C-Army tenant. They must be able to (a) see a tenant picker in the UI, (b) explicitly switch their session into the C-Army scope, and (c) have every action they take while switched be audit-logged with both their real identity and the impersonated tenant.

**Why this priority**: Same priority as US1 — without it, CSPs can't operate the platform.

**Independent Test**: Authenticate as a user with role `CSP.Admin`, exercise the tenant-switch endpoint, then call the same APIs as US1 and confirm responses now scope to the chosen tenant. Verify every request emits an audit row containing real user `oid`, real `tid`, and `ImpersonatedTenantId`.

**Acceptance Scenarios**:

1. **Given** a user with role `CSP.Admin`, **When** they call `GET /api/tenants`, **Then** they see all tenants in the deployment.
2. **Given** an active CSP-Admin session, **When** they call `POST /api/tenants/{tenantId}/impersonate`, **Then** subsequent requests in that session carry an `X-Impersonated-Tenant` header (or session cookie) and all queries scope to that tenant.
3. **Given** an active impersonation, **When** the user calls any read or write endpoint, **Then** an audit row is written with `actor.oid`, `actor.tid`, `effective.tenantId`, `actor.role=CSP.Admin`, and the action.
4. **Given** a non-CSP-Admin user, **When** they attempt the impersonation endpoint, **Then** the API returns 403 with `errorCode: FORBIDDEN_NOT_CSP_ADMIN`.
5. **Given** an impersonation session, **When** the user explicitly ends it (`DELETE /api/tenants/impersonation` or session expiry), **Then** the next request scopes back to the user's home tenant.

### User Story 3 — Single-Tenant Deployment Continues to Work Unchanged (Priority: P1)

A mission owner who self-hosts the existing single-tenant Docker stack must be able to upgrade to this release without manual data migration steps and without seeing tenant pickers, impersonation UI, or any extra friction.

**Why this priority**: We cannot break the existing self-host install path used by current mission-owner customers.

**Independent Test**: Set `ATO_DEPLOYMENT__MODE=SingleTenant`, run an upgrade against an existing dev DB, and confirm: (a) every existing row is migrated to a single auto-provisioned `Tenant` row; (b) no tenant picker appears in the UI; (c) all existing endpoints behave identically.

**Acceptance Scenarios**:

1. **Given** `ATO_DEPLOYMENT__MODE=SingleTenant` and an existing database with rows lacking `TenantId`, **When** the application starts, **Then** a startup migration provisions a `default-tenant` row, backfills `TenantId` on every entity, and emits a single `Migrated N rows to default tenant` log line.
2. **Given** a single-tenant deployment, **When** a user logs in, **Then** the dashboard never shows a tenant picker, organization switcher, or CSP-Admin-only menu items.
3. **Given** the same deployment, **When** the operator switches `ATO_DEPLOYMENT__MODE` to `MultiTenant` and restarts, **Then** the existing data is now scoped to the default tenant and a tenant-picker becomes visible to users with the `CSP.Admin` role.

### User Story 4 — Tenant-Bounded Onboarding (Priority: P2)

When a new organization onboards, a CSP-Admin first pre-provisions the tenant via API, then the first end-user from that Entra tenant is routed into a **Tenant-and-Organization Onboarding Wizard** that captures tenant-level information (legal entity, DoD component, POC, HQ address, classification level, AO) and the first organization. Every wizard artifact (registrations, role assignments, document templates, narrative seeds, audit entries, SSP-PDF imports, eMASS imports) is bound to that tenant. A second tenant onboarding the same Azure subscription does not collide or see the first tenant's registrations.

**Why this priority**: Onboarding is the entry point for every new tenant. Existing onboarding entities already have `TenantId`, but the writes today derive that ID from request claims — we must verify all paths and replace any hardcoded `00000000-0000-0000-0000-000000000001` constants with the resolved tenant.

**Independent Test**: Two test tenants run the onboarding wizard in parallel against the same MCP instance. Each completes Step 1 (organization profile) through Step N (final SSP draft) without seeing each other's data. Same Azure subscription registered by both tenants creates two distinct `AzureSubscriptionRegistration` rows (one per tenant).

**Acceptance Scenarios**:

1. **Given** an empty deployment, **When** Tenant A completes the org-context step, **Then** a `Tenant` row and matching `OrganizationContext` row are created, both scoped to Tenant A's Entra `tid`.
2. **Given** Tenant A has registered subscription `sub-1234`, **When** Tenant B independently registers the same `sub-1234`, **Then** two rows exist in `AzureSubscriptionRegistrations` with distinct `(TenantId, SubscriptionId)` keys.
3. **Given** any onboarding wizard step, **When** an artifact is created (role assignment, narrative seed, doc template, …), **Then** its `TenantId` matches the requesting user's effective tenant.

### User Story 5 — Defense-in-Depth at the Database Layer (Priority: P2)

Even if a developer forgets a query filter, a misconfigured EF migration drops a filter, or an attacker bypasses application code by calling the SQL endpoint directly via a stolen connection string, the database **itself** must refuse cross-tenant reads and writes for non-CSP-Admin connections.

**Why this priority**: Security defense-in-depth. EF query filters are a guardrail; SQL Server Row-Level Security (RLS) is the wall.

**Independent Test**: Connect to the SQL Server with the application's normal connection (which does NOT carry a CSP-Admin session-context flag). Try to `SELECT *` from `RegisteredSystems` after setting `SESSION_CONTEXT('TenantId') = 'X'`. Verify only Tenant X rows return. Try to insert a row with a different `TenantId` — verify the security policy blocks it.

**Acceptance Scenarios**:

1. **Given** a SQL Server deployment, **When** the application connects with `SESSION_CONTEXT('TenantId') = 'A'`, **Then** raw `SELECT` against any tenant-scoped table returns only Tenant A rows.
2. **Given** the same connection, **When** the app issues `INSERT … VALUES (TenantId='B', …)`, **Then** the row is rejected by the RLS BLOCK predicate.
3. **Given** a SQL Server deployment, **When** the connection sets `SESSION_CONTEXT('IsCspAdmin') = 'true'`, **Then** reads and writes succeed across tenants (impersonation path).
4. **Given** the deployment uses SQLite (development), **When** RLS is unavailable, **Then** the same protections are emulated by EF query filters only and a startup warning is logged that RLS is not active.

### User Story 6 — Auditable Cross-Tenant Operations (Priority: P3)

Every CSP-Admin impersonation, every cross-tenant export, every administrative action that touches a tenant other than the actor's own home tenant must be persisted to a tamper-evident audit log searchable by tenant, by actor, and by time range.

**Why this priority**: Required for FedRAMP / DoD authorization but not strictly blocking on launch — the action will already be in the existing `AuditLogs` table; this story formalizes the schema and the export.

**Independent Test**: Trigger 5 impersonation cycles and 5 ordinary in-tenant actions. Run `GET /api/audit?tenantId=X&actorOid=Y&since=…`. Confirm all 10 actions appear with the correct fields and that filter predicates work.

**Acceptance Scenarios**:

1. **Given** a CSP-Admin impersonation, **When** any read or write endpoint executes, **Then** an `AuditLogEntry` is written with `ActorTenantId`, `EffectiveTenantId`, `ActorOid`, `Action`, `Resource`, `Outcome`, `CorrelationId`.
2. **Given** an existing audit log, **When** an authorized auditor queries `/api/audit`, **Then** they can filter by tenant, actor, action, and time range and get paginated results.

### User Story 7 — CSP First-Use Onboarding (Priority: P1)

A CSP operator stands up a new MultiTenant deployment of ATO Copilot. The very first time they sign in (with the `CSP.Admin` role), the dashboard routes them to a **CSP Onboarding Wizard** that captures the hosting CSP's own identity — legal entity, public display name (e.g., "Flankspeed"), logo, primary support contact, and default classification floor for hosted tenants. Until that wizard is complete, no tenant can be pre-provisioned, no per-tenant onboarding wizard runs, and ordinary tenant-scoped requests return `503 CSP_ONBOARDING_INCOMPLETE`. Once complete, the deployment is fully operational and the existing per-tenant onboarding (US4) becomes available.

**Why this priority**: A `MultiTenant` deployment cannot legally onboard customer tenants without first capturing the hosting CSP's own legal/contact metadata — audit, support, and customer communications all reference it. This blocks US2, US4, and US8 in `MultiTenant` mode.

**Independent Test**: Boot a fresh deployment with `ATO_DEPLOYMENT__MODE=MultiTenant` and an empty database. Sign in as a `CSP.Admin`. Verify the dashboard redirects to `/onboarding/csp` and every other tenant-scoped endpoint returns `503 CSP_ONBOARDING_INCOMPLETE`. Step through the wizard, submit it, and confirm `CspProfile.OnboardingState = Active`, that per-tenant onboarding (US4) is now reachable, and that re-running the wizard URL is idempotent (no duplicate row).

**Acceptance Scenarios**:

1. **Given** a fresh `MultiTenant` deployment with no `CspProfile` row, **When** a `CSP.Admin` signs in, **Then** they are routed to `/onboarding/csp` and every other tenant-scoped endpoint returns `503 CSP_ONBOARDING_INCOMPLETE`.
2. **Given** the CSP wizard is incomplete, **When** a non-`CSP.Admin` user signs in, **Then** they receive `503 CSP_ONBOARDING_INCOMPLETE` on every endpoint except `/health` and `/api/auth/*` (they cannot even reach `/api/onboarding/tenant/*`).
3. **Given** a `CSP.Admin` completes the wizard, **Then** `CspProfile.OnboardingState = Active` and `OnboardingCompletedAt` is set, an audit row `Action = CspOnboarding.Complete` is emitted, and per-tenant pre-provisioning (FR-053) and the per-tenant onboarding wizard (FR-054) become available.
4. **Given** `ATO_DEPLOYMENT__MODE = SingleTenant`, **When** the application boots, **Then** no `CspProfile` row is created, the CSP wizard is unreachable, and `/api/csp/onboarding/*` returns `404`.
5. **Given** an existing SingleTenant deployment is switched to `MultiTenant`, **When** the application boots, **Then** the CSP wizard appears (no `CspProfile` row exists yet) and existing tenant data is preserved but locked behind `503` until the CSP completes the wizard.
6. **Given** a CSP-Admin closes the browser mid-wizard, **When** they return, **Then** the wizard resumes at the last incomplete step and no duplicate state is created.

### User Story 8 — CSP Cross-Tenant Operational Dashboard (Priority: P2)

A Flankspeed CSP-Admin needs a single-screen operational view of every customer tenant they host: how many tenants are `Active` vs `Suspended` vs `Disabled`, how many systems and organizations live under each, how many ATOs are `Authorized` / `InProcess` / `Denied`, how many open findings/POA&Ms/deviations exist in aggregate. They must be able to drill from a tenant row directly into impersonation. They must NOT have to impersonate each tenant in turn just to count rows.

**Why this priority**: Operational. Without it, CSP-Admins cannot triage support load or report platform-wide metrics to leadership without writing custom SQL.

**Independent Test**: Seed 3 tenants × 5 systems × varied ATO statuses. As a `CSP.Admin`, call `GET /api/csp/dashboard/summary` and verify aggregate counts equal the sum across all tenants. Call `GET /api/csp/dashboard/tenants` and verify each tenant row reports the correct per-tenant KPI. Click a tenant row and verify the UI invokes `POST /api/tenants/{id}/impersonate` and routes into that tenant's standard dashboard with the impersonation banner visible.

**Acceptance Scenarios**:

1. **Given** 3 tenants × 5 systems × varied ATO statuses, **When** `CSP.Admin` calls `GET /api/csp/dashboard/summary`, **Then** the response includes total tenants by status, total organizations, total systems, ATOs grouped by `Authorized` / `InProcess` / `Denied`, open findings by severity (`Critical`/`High`/`Moderate`/`Low`), open POA&Ms by status, and open deviations by status — all aggregated cross-tenant.
2. **Given** the same data, **When** `CSP.Admin` calls `GET /api/csp/dashboard/tenants?page=1&pageSize=50`, **Then** the response is a paginated list with `tenantId`, `displayName`, `status`, `systemCount`, `atoStatusCounts`, `openFindingCount`, `lastActivityTimestamp` for each tenant.
3. **Given** a non-CSP-Admin user, **When** they call any `/api/csp/dashboard/*` endpoint, **Then** the API returns `403 FORBIDDEN_NOT_CSP_ADMIN`.
4. **Given** a tenant in `Disabled` status, **When** `CSP.Admin` views the all-up dashboard, **Then** the tenant appears with a `Disabled` badge but is excluded from active-KPI rollups (its rows are excluded from "open findings", "systems", etc.).
5. **Given** the all-up dashboard is loaded, **When** `CSP.Admin` clicks a tenant row, **Then** the UI calls `POST /api/tenants/{tenantId}/impersonate` and routes into that tenant's standard dashboard with the impersonation banner visible (US2 path).
6. **Given** the CSP onboarding wizard is incomplete, **When** any `/api/csp/dashboard/*` endpoint is called, **Then** the API returns `503 CSP_ONBOARDING_INCOMPLETE` (US7 gate).

### User Story 9 — CSP-Provided Components from Uploaded ATOs (Priority: P1, MultiTenant only)

During CSP onboarding (US7), a CSP-Admin uploads one or more existing ATO artifacts (e.g., the CSP's FedRAMP SSP PDF, an eMASS ZIP, an OSCAL JSON SSP, a SAR/POAM workbook). The system parses each artifact and extracts component-like entries (system descriptions, services, infrastructure layers) as **CSP-inherited components** that live in the system tenant and are visible read-only to **every** hosted tenant. For each extracted component, the system attempts to generate `Capability` rows mapped to NIST 800-53 controls; capabilities the AI cannot map with confidence are still persisted but flagged `NeedsReview` with a reason. The CSP-Admin sees a Mapped vs. NeedsReview tally before finalizing submit, and any hosted tenant can immediately reference the resulting CSP-inherited components when authoring systems and control inheritance.

**Why this priority**: This is the core mechanism by which a hosting CSP delivers inheritance value to its tenants. Without it, every hosted mission owner re-authors the same shared infrastructure components from scratch, losing the primary reason to use a CSP in the first place. The auto-mapping → NeedsReview path is what makes the inheritance actionable rather than a pile of opaque text.

**Independent Test**: Boot a fresh `MultiTenant` deployment, sign in as `CSP.Admin`, upload a sample SSP PDF in the CSP Onboarding Wizard, and complete submit. Verify that (a) at least one `CspInheritedComponent` row is created in the system tenant; (b) each component has ≥ 1 `CspInheritedCapability` row; (c) capabilities with low AI confidence are flagged `NeedsReview` with a non-empty `MappingFailureReason`; (d) the wizard's review step shows a `Mapped: N / NeedsReview: M` tally; (e) after onboarding completes, a Mission Owner signed into a freshly-pre-provisioned tenant can list the CSP-inherited components via `GET /api/csp/inherited-components` and reference them when authoring an inheritance default — without being able to write to them.

**Acceptance Scenarios**:

1. **Given** a CSP-Admin in the ATO Documents step of the wizard, **When** they upload a PDF SSP via `POST /api/csp/onboarding/atos/upload` (multipart), **Then** the file is parsed, candidate components are extracted, and the response returns the count of `componentsExtracted`, `capabilitiesMapped`, `capabilitiesNeedsReview` plus a structured preview list.
2. **Given** an uploaded eMASS ZIP, **When** parsing succeeds, **Then** components and POA&M-derived capabilities are extracted from the embedded SSP / SAR / POAM artifacts and stored under `CspInheritedComponent.Source = "eMASS:<filename>"`.
3. **Given** an unsupported file type or a corrupt file, **When** the upload is processed, **Then** the API returns `400 UNSUPPORTED_ATO_DOCUMENT` (or `400 PARSE_FAILED` with details) and no component is created; other previously uploaded documents in the same wizard session are NOT affected.
4. **Given** a successfully parsed component, **When** the AI capability-mapping pipeline runs, **Then** capabilities mapped with confidence ≥ the configured threshold are stored with `Status = Mapped` and a JSON list of NIST control IDs; capabilities below the threshold OR where the AI errors out are stored with `Status = NeedsReview` and a non-empty `MappingFailureReason`.
5. **Given** the wizard is in the Review step, **When** the CSP-Admin views the summary, **Then** they see `Components: N`, `Capabilities Mapped: M`, `Capabilities Needs Review: K` with a link to a per-component detail panel listing every `NeedsReview` capability and its reason.
6. **Given** CSP onboarding has completed and a Mission Owner is signed into Tenant A, **When** they call `GET /api/csp/inherited-components`, **Then** they see all `Status = Published` CSP-inherited components and may reference them when authoring an inheritance default for any system in Tenant A; attempts to `POST` / `PUT` / `DELETE` against `/api/csp/inherited-components/*` from a non-CSP-Admin context return `403 FORBIDDEN_NOT_CSP_ADMIN`.
7. **Given** a `CspInheritedCapability` flagged `NeedsReview`, **When** a CSP-Admin calls `PATCH /api/csp/inherited-components/{componentId}/capabilities/{capabilityId}/review` with mapped controls and an optional override note, **Then** the row transitions to `Status = Mapped`, an audit row `Action = CspInheritedCapability.Review` is emitted, and the change is visible to all tenants on next read.
8. **Given** a CSP-Admin uploads a 60 MB file, **When** the upload exceeds the 50 MB limit, **Then** the API returns `413 ATO_DOCUMENT_TOO_LARGE` and no temp file is retained.

### User Story 10 — Mission Owner Consumes a CSP-Inherited Capability (Priority: P1, MultiTenant only)

A Coastal Watch mission owner is authoring an SSP for a system hosted on Flankspeed. While completing control inheritance for `IA-2` and `IA-2(1)`, they pick a CSP-published `CspInheritedCapability` from the `CspInheritedComponent` catalog. The platform must (a) treat the CSP's uploaded SSP / OSCAL artifact as the evidence source for those two controls inside Coastal's tenant; (b) generate Coastal's control narrative using the CSP component description + capability description + source-artifact reference as contextual input — the wording must explicitly state the control is inherited from "Flankspeed" and cite the source document by name; (c) when the AI narrative service is unreachable, write a deterministic stub narrative that still names the CSP and source so the SSP draft is never blank, mark it `Status = NeedsReview`, and remain regenerable via the existing narrative-regenerate endpoint; (d) reuse the existing inheritance / evidence / narrative services from Features 008, 024, 038, 043, 044 — no parallel implementations, no new endpoints.

**Why this priority**: This is what makes US9's published catalog *actionable*. Without consumption-side wiring, US9 produces read-only rows that cannot be attached to a tenant's controls, defeating the purpose. Together US9 + US10 are the inheritance value loop and both are required for a `MultiTenant` deployment to deliver day-one value to its hosted tenants.

**Independent Test**: Seed Tenant A (Coastal) with one system. The CSP has US9-published a `CspInheritedCapability` covering `AC-2` and `AC-2(1)`. See <SourceFileName>. Capability description: <Capability.Description>."` is persisted with `Status = NeedsReview` (Feature 024 status field), and a later call to the existing narrative-regenerate endpoint produces an AI narrative without manual intervention.
4. **Given** a Mission Owner who is not CSP-Admin, **When** they view the inherited control card on the dashboard, **Then** the card renders `Source: <CspProfile.DisplayName> (Inherited from CSP)` per FR-083 and a link to the source document via `CspInheritedComponent.SourceArtifactReference`.

### Edge Cases

- **Bootstrapping**: First-ever startup with empty DB. Must create a system tenant `Ato.Copilot.System` for cross-cutting infra rows (e.g., NIST control catalog, OSCAL framework metadata) that are not tenant-scoped. The CAC middleware must allow unauthenticated calls only for `/health`; everything else must resolve to a tenant or fail.
- **CSP-bootstrap gate (MultiTenant only)**: Until the CSP-level onboarding wizard (US7) is complete, the only authenticated endpoints that succeed are `/api/csp/onboarding/*` (CSP-Admin only) and `/api/auth/*`. All other tenant-scoped endpoints return `503 CSP_ONBOARDING_INCOMPLETE`.
- **Partial ATO-document parse (US9)**: A wizard upload may succeed for the PDF text-extraction step but fail for the AI component-extraction step (e.g., model rate-limit, transient outage). The endpoint MUST still record an audit entry, MUST NOT corrupt the wizard state, MUST surface the failure to the CSP-Admin with a retry option, and MUST NOT prevent the CSP-Admin from completing onboarding without that document — the CSP-Admin can revisit the ATO Documents step after onboarding via `POST /api/csp/inherited-components/import` (CSP-Admin only).
- **AI mapping unavailable (US9)**: When the AI capability-mapping service is unavailable (no Azure OpenAI, network blocked, quota exceeded), extracted components MUST still be persisted with **zero** capabilities and the wizard MUST display a banner explaining that capability mapping was skipped; the CSP-Admin can re-run mapping later via `POST /api/csp/inherited-components/{id}/remap`.
- **NIST control catalog & framework data** must be readable by every tenant (it's reference data, not customer data). These rows live under `Tenant = '<system>'` and bypass the query filter via a `[GlobalReference]` marker on the entity.
- **Failed claim resolution**: Token has no `tid` claim and no `X-Tenant-Id` header. Default behavior: 401 `MISSING_TENANT_CLAIM`. (Not 500, so we don't expose internals.)
- **Stale impersonation cookie after CSP-Admin role revoked**: Every request re-evaluates the role; if the role is gone the cookie is ignored and the user falls back to their home tenant.
- **Cross-tenant FK reference**: A `ComponentSystemAssignment` row must never reference a `RegisteredSystem` from a different tenant. EF SaveChanges interceptor validates `(this.TenantId == referenced.TenantId)` before commit.
- **Cross-tenant inheritance / shared evidence**: All inheritance and evidence-share FKs (e.g., `OrgInheritanceDefault`, `EvidenceShare`, control inheritance mappings introduced by Features 043/044/038) MUST resolve within the same tenant. Cross-tenant inheritance is only possible by way of CSP-Admin-published `[GlobalReference]` baselines stored in the system tenant.
- **Consuming a `NeedsReview` CSP capability (US9 → US10)**: A tenant cannot bind a `CspInheritedCapability` whose `Status = NeedsReview` to an inheritance row. Any save attempt MUST return `409 CSP_CAPABILITY_NEEDS_REVIEW` (referencing FR-107) and the dashboard MUST show the capability as disabled-with-tooltip in the picker until a CSP-Admin completes review (FR-105). This prevents a tenant from inheriting evidence linked to an unverified control mapping.
- **MCP tool invoked from VS Code extension or Teams bot**: The Channels library carries the user's identity. Tools must read the same `ITenantContext` that endpoints use.
- **Bulk export at the SQL level**: A DBA running `BACKUP DATABASE` is out of scope for this feature — operational policy, not application policy.

## Requirements *(mandatory)*

### Functional Requirements

#### Data Model
- **FR-001**: System MUST add a `Tenants` table with `Id (Guid, PK)`, `EntraTenantId (Guid, unique nullable)`, `DisplayName`, `LegalEntityName`, `DoDComponent (nullable)`, `PrimaryPocName`, `PrimaryPocEmail`, `PrimaryPocPhone (nullable)`, `HqAddressLine1 (nullable)`, `HqAddressLine2 (nullable)`, `HqCity (nullable)`, `HqStateOrProvince (nullable)`, `HqPostalCode (nullable)`, `HqCountry (nullable)`, `DefaultClassificationLevel (Unclassified|CUI|Secret, default Unclassified)`, `AuthorizingOfficialName (nullable)`, `AuthorizingOfficialEmail (nullable)`, `TimeZone (default UTC)`, `Status (Active|Suspended|Disabled, default Active)`, `OnboardingState (Pending|InWizard|Active, default Pending)`, `CreatedAt`, `CreatedBy`. One row per customer-facing tenant boundary. Fields beyond `Id` / `EntraTenantId` / `DisplayName` / `Status` / `OnboardingState` MAY be populated by the onboarding wizard rather than by the pre-provisioning API.
- **FR-002**: System MUST add an `Organizations` table with `Id (Guid, PK)`, `TenantId (FK → Tenants.Id, indexed)`, `Name`, `Description`, `CreatedAt`. A tenant may have many organizations (e.g., divisions, mission programs).
- **FR-003**: System MUST add a `TenantId (Guid, NOT NULL, indexed)` column to **every** DbSet in `AtoCopilotContext` except those marked `[GlobalReference]` (NIST controls, framework metadata, OSCAL catalog, the `Tenants` table itself).
- **FR-004**: System MUST add an `OrganizationId (Guid, nullable, indexed)` column where the entity belongs to a sub-organization (e.g., `RegisteredSystem`, `OrgInheritanceDefault`, `EvidenceArtifact`). Null means tenant-level.
- **FR-005**: The existing `OrganizationContext` table MUST become a child of `Organizations` with FK; current `TenantId` column becomes the link to the new `Tenants.Id` (a value-conversion or backfill, since today it stores the Entra `tid` directly).
- **FR-006**: System MUST add a `CspProfile` table — a singleton (at most one row) that lives in the system tenant and is marked `[GlobalReference]`. Columns: `Id (Guid PK)`, `LegalEntityName`, `DisplayName` (shown to all hosted tenants, e.g., "Flankspeed"), `LogoUrl (nullable)`, `PrimarySupportEmail`, `SupportPhone (nullable)`, `DefaultClassificationFloor (Unclassified|CUI|Secret, default Unclassified)`, `OnboardingState (Pending|InWizard|Active, default Pending)`, `OnboardingCompletedAt (nullable)`, `CreatedAt`, `CreatedBy`, `UpdatedAt`, `UpdatedBy`, `RowVersion`. Created lazily — only `MultiTenant` deployments instantiate this row, and only when the first `CSP.Admin` signs in. `SingleTenant` deployments MUST NOT create a `CspProfile` row.
- **FR-007**: System MUST add a `CspInheritedComponents` table — a tenant-less catalog living in the system tenant and marked `[GlobalReference]`. Columns: `Id (Guid PK)`, `CspProfileId (FK → CspProfile.Id)`, `Name`, `Description (nvarchar(max))`, `ComponentType (e.g., Infrastructure|Platform|Service|Identity|Network|Storage|Compute, default Service)`, `SourceFileName (nullable)`, `SourceFormat (Pdf|Docx|OscalJson|Xlsx|EmassZip|Manual, default Manual)`, `SourceArtifactReference (nullable, nvarchar(2048))`, `Status (Draft|Published|Archived, default Draft)`, `ImportedAt`, `ImportedBy`, `UpdatedAt`, `UpdatedBy`, `RowVersion`. Visible read-only to every hosted tenant via `GET /api/csp/inherited-components`. Tenant-local entities (e.g., `RegisteredSystem`, `OrgInheritanceDefault`) MAY reference these by `Id` without violating FR-080 because the rows are `[GlobalReference]`.
- **FR-008**: System MUST add a `CspInheritedCapabilities` table — a child of `CspInheritedComponents`, also `[GlobalReference]`. Columns: `Id (Guid PK)`, `ComponentId (FK → CspInheritedComponents.Id)`, `Name`, `Description (nvarchar(max))`, `MappedNistControlIds (nvarchar(max), JSON array of strings, e.g., ["AC-2","AC-2(1)"])`, `MappingConfidence (decimal(4,3), 0.000–1.000, nullable)`, `Status (Mapped|NeedsReview, default NeedsReview)`, `MappingFailureReason (nvarchar(2048), nullable)`, `MappedBy (User|AI, default AI)`, `ReviewedBy (nullable)`, `ReviewedAt (nullable)`, `CreatedAt`, `UpdatedAt`, `RowVersion`. The wizard tally `Capabilities Mapped` counts rows where `Status = Mapped`; `Capabilities Needs Review` counts rows where `Status = NeedsReview`.

#### Tenant Resolution
- **FR-010**: Pipeline MUST add a `TenantResolutionMiddleware` that runs after `CacAuthenticationMiddleware` and before `ComplianceAuthorizationMiddleware`. It populates an `ITenantContext` scoped service with: `TenantId`, `OrganizationId?`, `IsCspAdmin`, `ImpersonatedTenantId?`, `EffectiveTenantId` (= ImpersonatedTenantId ?? TenantId).
- **FR-011**: Tenant resolution MUST honour, in order: (a) explicit `X-Impersonated-Tenant` header for CSP-Admins, (b) the user's `tid` Entra claim mapped to a `Tenants.EntraTenantId` row, (c) in `SingleTenant` deployment mode only, the singleton default tenant.
- **FR-012**: When tenant resolution fails (no claim, no header, no default), the middleware MUST short-circuit with `401 MISSING_TENANT_CLAIM` and emit a structured log without exposing internals.

#### Application-Level Isolation
- **FR-020**: `AtoCopilotContext.OnModelCreating` MUST register a `HasQueryFilter` on every tenant-scoped entity that filters `e.TenantId == _tenantContext.EffectiveTenantId` UNLESS `_tenantContext.IsCspAdmin && _tenantContext.ImpersonatedTenantId == null` (CSP-Admin in dashboard view sees all).
- **FR-021**: `AtoCopilotContext.SaveChangesAsync` MUST stamp `TenantId = _tenantContext.EffectiveTenantId` on every newly added tenant-scoped entity that has not already set a non-default value, and MUST reject saves where an entity's `TenantId` differs from the effective tenant (unless CSP-Admin override is asserted).
- **FR-022**: A new `[GlobalReference]` attribute MUST be defined and applied to entities that are reference data (e.g., `NistControl`, OSCAL framework metadata). The query filter MUST skip these.
- **FR-023**: Every dashboard endpoint group MUST add `.RequireAuthorization()` (already has it) and use `ITenantContext` for any explicit tenant predicate; raw `db.<DbSet>.Find(id)` calls MUST be replaced with calls that go through the query filter (e.g., `FirstOrDefaultAsync(x => x.Id == id)`).
- **FR-024**: All 130+ MCP tools MUST receive `ITenantContext` via DI and use it identically.

#### Database-Level Isolation
- **FR-030**: For SQL Server deployments, an EF migration MUST install a `SECURITY POLICY` per tenant-scoped table with a `FILTER PREDICATE` on `TenantId = CAST(SESSION_CONTEXT(N'TenantId') AS uniqueidentifier)` and a `BLOCK PREDICATE AFTER INSERT, UPDATE` enforcing the same.
- **FR-031**: The application's DB connection MUST set `SESSION_CONTEXT(N'TenantId', @effectiveTenantId)` on every connection borrowed from the pool. A `DbConnectionInterceptor` performs this.
- **FR-032**: When `IsCspAdmin && ImpersonatedTenantId IS NULL`, the connection MUST set `SESSION_CONTEXT(N'IsCspAdmin', N'true')` and the security policy MUST allow that bypass via `OR (SESSION_CONTEXT(N'IsCspAdmin') = N'true')`.
- **FR-033**: For SQLite (development) deployments, RLS MUST be skipped with a warning logged at startup. EF query filters provide isolation in dev.

#### Deployment Modes
- **FR-040**: A new `DeploymentOptions` class bound from `ATO_DEPLOYMENT__*` env vars MUST expose `Mode = SingleTenant | MultiTenant` (default `SingleTenant`) and `DefaultTenantId` (auto-generated GUID, persisted to DB).
- **FR-041**: In `SingleTenant` mode, the UI MUST hide tenant pickers, the impersonation endpoints MUST 404, and all unscoped requests MUST resolve to the default tenant.
- **FR-042**: In `MultiTenant` mode, the impersonation endpoints MUST be enabled and the dashboard MUST surface a tenant indicator in the header.

#### CSP-Admin Cross-Tenant
- **FR-050**: The role `CSP.Admin` MUST be granted via membership in a designated **Entra ID Security Group** whose object-ID is configured in `Auth:RoleClaimMappings:CSP.Admin`. The Entra app registration MUST request `groupMembershipClaims` so group object-IDs flow on every token, and the role-claim pipeline MUST translate any matching group GUID into the `CSP.Admin` role on `ClaimsPrincipal`. Membership changes MUST take effect on the next token refresh (no server-side cache beyond token lifetime). Users with this role MUST have access to a `GET /api/tenants` list endpoint.
- **FR-051**: `POST /api/tenants/{tenantId}/impersonate` MUST verify the caller is `CSP.Admin`, that the target tenant exists, then issue a session cookie / header containing the impersonation. Impersonation MUST expire after 1 hour and MUST be revocable via `DELETE /api/tenants/impersonation`.
- **FR-052**: While impersonating, every audit row MUST carry both `ActorTenantId` (the CSP-Admin's home) and `EffectiveTenantId` (the impersonated tenant).

#### Tenant Provisioning & Onboarding
- **FR-053**: `POST /api/tenants` (CSP-Admin only) MUST pre-provision a `Tenants` row from a request body containing at minimum `EntraTenantId` and `DisplayName`. The row MUST be created with `Status = Active` and `OnboardingState = Pending`. The endpoint MUST emit an audit row with the actor's identity and the new `TenantId`, and MUST be idempotent on `EntraTenantId` (return 200 + existing row if already provisioned).
- **FR-054**: When a user signs in with an Entra `tid` that resolves to a `Tenants` row whose `OnboardingState != Active`, the dashboard MUST route them into the **Tenant-and-Organization Onboarding Wizard** before any other tenant-scoped API call succeeds. The wizard MUST capture, in order: (a) tenant-level fields (`LegalEntityName`, `DoDComponent`, `PrimaryPocName/Email/Phone`, HQ address, `DefaultClassificationLevel`, `AuthorizingOfficialName/Email`, `TimeZone`); (b) the first `Organization` (`Name`, `Description`). On submit it MUST update the `Tenants` row, create the `Organization`, and set `OnboardingState = Active`.
- **FR-055**: When a user signs in with an Entra `tid` that has **no** matching `Tenants.EntraTenantId` row, the system MUST NOT silently auto-provision a tenant. Behavior MUST be controlled by `Tenants:AllowSelfOnboarding` (default `false`): when `false`, the request MUST fail with `401 TENANT_NOT_PROVISIONED`; when `true`, a `Tenants` row MUST be created with `OnboardingState = InWizard` and the user routed into the wizard described in FR-054.
- **FR-056**: The onboarding wizard MUST be re-entrant (a user who closes the browser mid-wizard returns to the same step on next sign-in) and MUST emit audit rows on every step submission with `Action = TenantOnboarding.<StepName>`.
- **FR-057**: When `Tenants.Status = Active`, all reads and writes scoped to that tenant proceed normally.
- **FR-058**: When `Tenants.Status = Suspended`, the resolution middleware MUST allow read endpoints (HTTP `GET`, `HEAD`, `OPTIONS`) and idempotent MCP read tools to succeed, MUST reject write endpoints (`POST`, `PUT`, `PATCH`, `DELETE`) and write MCP tools with `423 TENANT_SUSPENDED`, and MUST still permit CSP-Admin impersonation into the tenant for support and audit access.
- **FR-059**: When `Tenants.Status = Disabled`, the resolution middleware MUST reject all requests for that tenant with `401 TENANT_DISABLED` (including reads), MUST exclude the tenant from non-CSP-Admin tenant-list responses, and MUST keep all underlying data rows intact for audit retention. CSP-Admin users MUST still be able to read the tenant's data via impersonation for offboarding and audit purposes. Status transitions (`Active` ↔ `Suspended` ↔ `Disabled`) MUST be performed via `PATCH /api/tenants/{tenantId}/status` (CSP-Admin only) and MUST emit an audit row capturing the old status, new status, actor, and reason.

#### Audit
- **FR-060**: Every dashboard request MUST add a `TenantId`, `EffectiveTenantId`, and `ImpersonatedTenantId?` field to the audit log row produced by `AuditLoggingMiddleware`.
- **FR-061**: A `GET /api/audit` endpoint (CSP-Admin only) MUST allow filtering by tenant id, actor, action, time range, with pagination.

#### Migration & Backwards Compat
- **FR-070**: Startup MUST create the system tenant (`Id = '00000000-0000-0000-0000-000000000000'`, `DisplayName = 'Ato.Copilot.System'`) on first boot for global reference rows.
- **FR-071**: Startup MUST detect missing-tenant rows (post-migration, pre-backfill) and either: (a) in SingleTenant mode, backfill them all to the default tenant; (b) in MultiTenant mode, fail-fast with an error log instructing the operator to run the migration utility described in FR-073–FR-076.
- **FR-072**: Existing `00000000-0000-0000-0000-000000000001` references in seed scripts and tests MUST be replaced with the resolved default tenant.
- **FR-073**: A built-in admin endpoint `POST /api/admin/migrate-to-multitenant` (CSP-Admin only) MUST accept `{ defaultTenantId: Guid, overrides?: [{ tableName: string, rowIdPrefix?: string, tenantId: Guid }] }` and perform an idempotent transactional backfill: for every tenant-scoped table with rows whose `TenantId IS NULL`, assign `TenantId = defaultTenantId` unless an override matches, then install the SQL Server security policies described in FR-030. The endpoint MUST be a no-op when no `NULL` `TenantId` rows remain.
- **FR-074**: A pre-flight endpoint `GET /api/admin/migrate-to-multitenant/preview` (CSP-Admin only) MUST return per-table counts of `(TotalRows, RowsMissingTenant, RowsAssignedByOverride, RowsAssignedToDefault)` without modifying any data.
- **FR-075**: A standalone `ato-cli` dotnet-tool MUST be published with at least the commands `ato-cli tenant default --id <guid>`, `ato-cli tenant assign --csv mapping.csv`, and `ato-cli tenant migrate --connection-string <cs> --default-tenant-id <guid>`. The CLI MUST connect directly to the database (no running app required), perform the same logic as FR-073/FR-074, and emit a JSON report of changes for offline / air-gapped operations.
- **FR-076**: Both the admin endpoint (FR-073) and the CLI (FR-075) MUST emit equivalent audit rows describing the migration (`Action = Tenant.Migrate`, `Resource = <tableName>`, `RowsAffected`, `Actor`, `CorrelationId`) and MUST be re-runnable without producing duplicate effects.

#### Cross-Tenant Sharing & Inheritance
- **FR-080**: All inheritance, share, and reference FKs introduced by prior features (including `OrgInheritanceDefault`, control inheritance mappings from Features 043/044, and evidence shares from Feature 038) MUST be tenant-local: the SaveChanges interceptor MUST reject any save where the row's `TenantId` differs from the referenced row's `TenantId`, except where the referenced row is marked `[GlobalReference]`.
- **FR-081**: A new `IsPublishable` capability MUST be added to control narratives, evidence artifacts, and inheritance defaults that a CSP wishes to publish across tenants. Publishing is performed only by `CSP.Admin` via `POST /api/global-baselines/publish` and creates a copy of the source row in the system tenant marked `[GlobalReference]`. The original tenant-local row remains owned by its tenant; the published copy is read-only to all tenants.
- **FR-082**: A CSP-Admin MUST be able to `DELETE /api/global-baselines/{id}` to unpublish a previously published baseline. Unpublishing MUST NOT affect tenant-local rows that were inherited from the published baseline; instead those tenants MUST be flagged for re-evaluation via an audit event.
- **FR-083**: The dashboard MUST show, on any inherited control or shared evidence, a clear source label using one of three canonical patterns so users cannot confuse a tenant-local inheritance with a CSP-published global baseline: (a) `Source: <CspProfile.DisplayName> (Inherited from CSP)` when the row is sourced from a `CspInheritedCapability` (FR-106); (b) `Source: Global Baseline` when the row is a CSP-Admin-published `[GlobalReference]` baseline (FR-081); (c) `Source: <Tenant.DisplayName>` when the row is a tenant-local inheritance from another organization in the same tenant. The same label MUST be emitted by document exports (SSP, SAR) so the source provenance survives outside the dashboard.

#### CSP Onboarding (MultiTenant only)
- **FR-090**: When `Mode = MultiTenant` and no `CspProfile` row has `OnboardingState = Active`, the resolution middleware MUST gate the deployment as follows: `CSP.Admin` users see only `/api/csp/onboarding/*`, `/api/auth/*`, and `/health` succeed (everything else returns `503 CSP_ONBOARDING_INCOMPLETE`); non-`CSP.Admin` users receive `503 CSP_ONBOARDING_INCOMPLETE` on every tenant-scoped endpoint and `/api/auth/*` succeeds for sign-in only. This gate runs BEFORE the per-tenant resolution gates (FR-054, FR-055).
- **FR-091**: A wizard at `/onboarding/csp` (CSP-Admin only) MUST capture, in order: (a) `LegalEntityName`, `DisplayName`, optional `LogoUrl`; (b) `PrimarySupportEmail`, optional `SupportPhone`; (c) `DefaultClassificationFloor`; (d) review + submit. The endpoints under `/api/csp/onboarding/*` (`GET /state`, `POST /identity`, `POST /support`, `POST /classification`, `POST /submit`) MUST be re-entrant: a CSP-Admin closing the browser mid-wizard returns to the same step on next sign-in.
- **FR-092**: On wizard submit, system MUST set `CspProfile.OnboardingState = Active`, persist `OnboardingCompletedAt`, and emit an audit row `Action = CspOnboarding.Complete` carrying the actor's identity and the resulting `CspProfile.Id`. Subsequent calls to `/api/csp/onboarding/submit` MUST return `409 CSP_ALREADY_ONBOARDED` rather than mutating the existing row.
- **FR-093**: `Mode = SingleTenant` deployments MUST NOT initialize a `CspProfile` row, and `/api/csp/onboarding/*` endpoints MUST return `404` so the wizard is unreachable. Switching `Mode` from `SingleTenant` to `MultiTenant` and restarting MUST surface the wizard on next CSP-Admin sign-in (no `CspProfile` row exists yet).

#### CSP Cross-Tenant Operational Dashboard
- **FR-094**: A `GET /api/csp/dashboard/summary` endpoint (CSP-Admin only) MUST return aggregate KPIs across **all** tenants in a single response: total tenants by status (`Active`/`Suspended`/`Disabled`), total organizations, total systems, ATOs grouped by `Authorized`/`InProcess`/`Denied` (read from `AuthorizationDecision` rows), open findings grouped by severity (`Critical`/`High`/`Moderate`/`Low`), open POA&Ms by status, and open deviations by status. The query MUST execute on the CSP-Admin global-view path (no impersonation cookie required) and MUST use the new composite `(TenantId, …)` indexes for performance.
- **FR-095**: A `GET /api/csp/dashboard/tenants` endpoint (CSP-Admin only) MUST return a paginated list of tenants (`page`, `pageSize`, default `50`, max `200`) with per-tenant KPIs (`tenantId`, `displayName`, `status`, `onboardingState`, `organizationCount`, `systemCount`, `atoStatusCounts`, `openFindingCount`, `lastActivityTimestamp`). MUST support sort by `displayName`, `status`, `openFindingCount`, `lastActivityTimestamp` and filter by `status`.
- **FR-096**: A `GET /api/csp/dashboard/atos` endpoint (CSP-Admin only) MUST return a paginated cross-tenant list of `AuthorizationDecision` rows joined with their owning tenant's `displayName`, supporting filter by `decisionStatus`, `decisionType`, and time range, sorted by `decisionDate` desc.
- **FR-097**: All `/api/csp/dashboard/*` endpoints MUST short-circuit with `503 CSP_ONBOARDING_INCOMPLETE` if `CspProfile.OnboardingState != Active` (US7 gate), and MUST return `403 FORBIDDEN_NOT_CSP_ADMIN` for non-CSP-Admin users.
- **FR-098**: Tenants in `Status = Disabled` MUST appear in `/api/csp/dashboard/tenants` with a `Disabled` indicator but MUST be excluded from the rollup counts in `/api/csp/dashboard/summary` (`organizationCount`, `systemCount`, `atoStatusCounts`, `openFindingCount`, etc.). The `summary` response MUST include a separate `disabledTenantCount` field for transparency.

#### CSP-Inherited Components from Uploaded ATOs (US9, MultiTenant only)
- **FR-099**: A new wizard step — **ATO Documents** — MUST be inserted between the existing **Classification** step and the **Review** step of the CSP Onboarding Wizard (FR-091). The step MUST accept multi-file uploads via `POST /api/csp/onboarding/atos/upload` (multipart/form-data) for the following content types: `application/pdf`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `application/json` (OSCAL SSP), `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` (FedRAMP SSP / SAR / POAM workbooks), `application/zip` (eMASS package). Maximum file size MUST be 50 MB; uploads exceeding the limit MUST return `413 ATO_DOCUMENT_TOO_LARGE`. The step is **optional** — a CSP MAY skip it and onboard without uploads, then add documents later via FR-104.
- **FR-100**: For each uploaded file, the system MUST select an appropriate parser (PdfPig for PDF — reusing the parser from Feature 047; `DocumentFormat.OpenXml` for DOCX; the existing OSCAL parser from Feature 022 for OSCAL JSON; `ClosedXML` for XLSX; `System.IO.Compression` to enumerate eMASS ZIP entries before delegating each entry to its format-specific parser), extract candidate component records (name, description, type), and persist them as `CspInheritedComponent` rows with `Status = Draft`, `SourceFileName = <uploaded name>`, and `SourceFormat` set accordingly. Parser failures MUST return `400 PARSE_FAILED` with a `details` field naming the failing parser; unsupported MIME types MUST return `400 UNSUPPORTED_ATO_DOCUMENT`.
- **FR-101**: For each `CspInheritedComponent` created in FR-100, the system MUST invoke the existing AI capability-mapping pipeline (`ICapabilityMappingService` from Feature 045 / Feature 008) to generate one or more candidate capabilities per component. The pipeline output is normalized into `CspInheritedCapability` rows whose `MappedNistControlIds` is a JSON array of NIST 800-53 control IDs and `MappingConfidence` is the model's per-mapping confidence score in `[0.0, 1.0]`. A configurable threshold `Csp:Inheritance:MappingConfidenceThreshold` (default `0.6`) MUST decide whether the row is stored with `Status = Mapped` (≥ threshold) or `Status = NeedsReview` (< threshold). Rows where the AI returned no controls at all MUST be stored with `Status = NeedsReview`, `MappedNistControlIds = "[]"`, and a `MappingFailureReason` of `"AI returned no candidate controls"`.
- **FR-102**: When the AI capability-mapping service is **unavailable** — meaning a transport-level failure such as HTTP 5xx, network timeout, no model configured, or quota exhausted **before** any candidate response is returned — the system MUST still persist `CspInheritedComponent` rows from FR-100 (so the parsing work is not lost) but MUST NOT create `CspInheritedCapability` rows automatically. The endpoint response MUST include `aiMappingAvailable = false` and a non-empty `aiMappingFailureReason`. The CSP-Admin can later re-run mapping via `POST /api/csp/inherited-components/{componentId}/remap` (CSP-Admin only). The wizard's Review step (FR-091.d) MUST display the `aiMappingAvailable = false` banner so the operator is not surprised by missing capabilities. **Distinction from FR-101**: FR-101 covers the case where the AI service responded successfully but the response was empty or low-confidence — those rows are created with `Status = NeedsReview`. FR-102 covers the case where the AI service did not produce a response at all — zero rows are created, and `aiMappingAvailable = false` is reported to the wizard.
- **FR-103**: On every wizard re-entry while the **ATO Documents** step is current, the response from `GET /api/csp/onboarding/state` MUST include an `atoDocuments` summary block with `documentsUploaded`, `componentsExtracted`, `capabilitiesMapped`, `capabilitiesNeedsReview`, and `aiMappingAvailable`. The `Review` step MUST surface the same tallies.
- **FR-104**: After CSP onboarding completes (`CspProfile.OnboardingState = Active`), CSP-Admins MUST still be able to import additional ATO documents and produce more `CspInheritedComponent` / `CspInheritedCapability` rows via `POST /api/csp/inherited-components/import` (multipart, same MIME / size rules as FR-099). This endpoint MUST also be the path used by the wizard's **ATO Documents** step internally, so behaviour is identical pre- and post-onboarding.
- **FR-105**: A `GET /api/csp/inherited-components` endpoint MUST return all `CspInheritedComponent` rows with `Status = Published` to **any** authenticated user across **any** tenant (read-only). A `GET /api/csp/inherited-components/{id}/capabilities` endpoint MUST return the component's capabilities including their `Status` and (for `NeedsReview`) `MappingFailureReason`. CSP-Admins additionally MUST see `Status = Draft` and `Status = Archived` rows. Mutation endpoints (`PATCH`, `DELETE`, `POST .../publish`, `POST .../archive`, `POST .../remap`, `PATCH .../capabilities/{capabilityId}/review`) MUST require `CSP.Admin` and MUST return `403 FORBIDDEN_NOT_CSP_ADMIN` otherwise. CSP-Admin reviews of `NeedsReview` capabilities MUST emit an audit row `Action = CspInheritedCapability.Review` with `ActorOid`, `ComponentId`, `CapabilityId`, prior `MappedNistControlIds`, new `MappedNistControlIds`, and the reviewer's note.
- **FR-106**: When a tenant-local entity (`OrgInheritanceDefault`, `ControlInheritanceMapping`, `RegisteredSystem.ComponentReferences`, etc.) references a `CspInheritedComponent` by `Id`, the FK reference MUST NOT trigger the cross-tenant FK rejection in FR-080 because `CspInheritedComponent` is `[GlobalReference]`. The dashboard MUST label these references with `Source: <CspProfile.DisplayName> (Inherited from CSP)` per FR-083 (canonical pattern (a)) so users distinguish CSP-inherited components from tenant-local components.

#### CSP-Inherited Capability Consumption (US10, MultiTenant only)
- **FR-107**: When a tenant-local entity (`OrgInheritanceDefault` from Feature 044, `ControlInheritanceMapping` from Feature 043, or any other inheritance bookkeeping introduced by prior features) is created or updated to reference a `CspInheritedCapability` (via a new nullable `SourceCspCapabilityId Guid?` FK on those entities, plus a denormalized `SourceCspComponentId Guid?` for query performance), the system MUST persist exactly one `EvidenceArtifact` row in the **consuming tenant** per control listed in `CspInheritedCapability.MappedNistControlIds`. Each row MUST have `Type = CspInheritedReference` (a new enum value added to Feature 038's `EvidenceArtifactType`) with structured payload `{ SourceCspComponentId, SourceCspCapabilityId, SourceFileName, SourceArtifactReference, IsImmutableSource = true }`. Persistence MUST go through the existing `IEvidenceArtifactService` from Feature 038 — **no new persistence service**. The `EvidenceArtifact` row is tenant-local so the consuming tenant can supplement it with their own annotations, but the underlying CSP source remains read-only because the referenced `CspInheritedComponent` is `[GlobalReference]`. **Domain-event contract**: Side-effects (the per-control `EvidenceArtifact` writes in FR-107 and the per-control narrative invocations in FR-108 / FR-109) MUST be triggered by a single `CspCapabilityConsumed { TenantId, CspProfileId, CspCapabilityId, CspComponentId, MappedControlIds[], Actor, OccurredAt }` domain event published by the existing `IOrgInheritanceDefaultService.SaveAsync` path through the existing `IDomainEventDispatcher` (synchronous, in-process; same pattern as Features 024 / 044). Delivery semantics MUST be at-most-once and MUST execute inside the same `SaveChangesAsync` transaction as the originating inheritance row so a failure rolls back both the inheritance row and the side-effects atomically. Re-saving the same inheritance row with no FK change MUST be idempotent (no duplicate evidence rows, no duplicate narrative invocations); the `CspCapabilityConsumptionHandler` MUST detect existing `(TenantId, ControlId, SourceCspCapabilityId)` evidence rows and skip them.
- **FR-108**: On the same consumption event (FR-107), the system MUST invoke the existing `IControlNarrativeService` from Features 008 / 024 once per control in `MappedNistControlIds` with a request whose **CspContext** block carries (a) `CspProfile.DisplayName`; (b) `CspInheritedComponent.Name` and `Description`; (c) `CspInheritedCapability.Description`; (d) `SourceFileName` + `SourceArtifactReference`; (e) the consuming tenant context. The existing prompt template MUST be extended with a single new CSP-context fragment — **no new prompt template family, no new narrative service**. The generated narrative MUST explicitly state the control is inherited from `<CspProfile.DisplayName>` and cite the source document by name. Output MUST be persisted via the existing narrative-save path (Feature 024).
- **FR-109**: When `IControlNarrativeService` is unavailable (HTTP error, AI quota exceeded, no model configured), the system MUST still complete FR-107 (the evidence link persists) and MUST seed each affected control narrative with the deterministic stub `"This control is inherited from <CspProfile.DisplayName> via the <CspInheritedComponent.Name> capability. See <SourceFileName>. Capability description: <Capability.Description>."`. The narrative MUST be marked `Status = NeedsReview` using the existing Feature 024 narrative-governance status field. A subsequent AI regeneration MUST be possible via the existing narrative-regenerate endpoint — **no new endpoint** is permitted.

### Key Entities

- **Tenant** — the root authorization boundary. Mapped 1:1 to an Entra tenant for SSO purposes. Owns Organizations, Users, and all data.
- **Organization** — a sub-grouping inside a Tenant (e.g., a mission program or division). Used by `OrganizationContext` and as the optional grouping for `RegisteredSystem` etc.
- **CspProfile** — singleton entity (at most one row) representing the hosting CSP itself. Lives in the system tenant; marked `[GlobalReference]`. Created only in `MultiTenant` mode and only after the first CSP-Admin completes the CSP Onboarding Wizard (US7). Drives branding/audit/support metadata for all hosted tenants.
- **CspInheritedComponent** — a component sourced from a CSP-uploaded ATO artifact (PDF SSP, OSCAL JSON, eMASS ZIP, etc.) during CSP onboarding (US9) or via post-onboarding import. Lives in the system tenant; marked `[GlobalReference]`. Visible read-only to every hosted tenant. Owned by `CspProfile`; carries source-file provenance and a `Status` lifecycle (`Draft → Published → Archived`).
- **CspInheritedCapability** — a capability of a `CspInheritedComponent` produced by the AI mapping pipeline. Holds the JSON list of mapped NIST 800-53 control IDs and the AI's confidence score. Capabilities below the configured confidence threshold (or where the AI returned nothing) are persisted with `Status = NeedsReview` and a free-text `MappingFailureReason` so a CSP-Admin can complete the mapping later via `PATCH /api/csp/inherited-components/{id}/capabilities/{capabilityId}/review`.
- **TenantContext (runtime)** — the resolved per-request scope. Carries `TenantId`, `OrganizationId`, `IsCspAdmin`, `ImpersonatedTenantId`, `EffectiveTenantId`. Implemented as an `IScoped` DI service.
