## Feature 027 — CAC Simulation Mode

Enables local development and integration testing of CAC-authenticated workflows without a physical smart card.

### Highlights

- **Simulation middleware**: `CacAuthenticationMiddleware` branches to a simulated `ClaimsPrincipal` when `CacAuth:SimulationMode=true`
- **Configurable identity**: UPN, display name, certificate thumbprint, and roles defined in `CacAuth:SimulatedIdentity` config section
- **Environment variable overrides**: `CacAuth__SimulatedIdentity__UserPrincipalName`, `CacAuth__SimulatedIdentity__SimulatedRoles__0`, etc.
- **Audit trail safety**: Simulated sessions are flagged to prevent contamination of compliance evidence
- **Development-only guard**: Simulation mode is restricted to Development environment — fails fast in Production

### Configuration

```json
{
  "CacAuth": {
    "SimulationMode": true,
    "SimulatedIdentity": {
      "UserPrincipalName": "dev.user@mil",
      "DisplayName": "Dev User",
      "CertificateThumbprint": "ABC123...",
      "SimulatedRoles": ["Global Reader", "SharePoint Administrator"]
    }
  }
}
```

## Feature 028 — Azure AI Foundry Agent Integration

Adds Azure AI Foundry Agents as a parallel AI processing path alongside the existing IChatClient pipeline.

### Highlights

- **PersistentAgentsClient registration**: Factory-based DI registration from `AzureAi:FoundryProjectEndpoint` config
- **Agent provisioning**: Compliance, Configuration, and KnowledgeBase agents auto-created (or reused) in Azure AI Foundry at startup with correct system prompts and tool schemas
- **Thread/run lifecycle**: Server-side thread management with run polling, tool dispatch loops, and automatic cleanup
- **Dual-path fallback**: `AzureAi:Provider` selects Foundry or OpenAI IChatClient — automatic fallback from Foundry → IChatClient → deterministic routing
- **Azure Government support**: Correctly targets Azure Government AI Foundry endpoints
- **Configurable timeouts**: `RunTimeoutSeconds` (default 60), `MaxToolIterations` (default 10), `Temperature` (default 0.3)

### Configuration

```json
{
  "AzureAi": {
    "Enabled": true,
    "Provider": "Foundry",
    "FoundryProjectEndpoint": "https://<project>.services.ai.azure.us/",
    "DeploymentName": "gpt-4o",
    "RunTimeoutSeconds": 60,
    "MaxToolIterations": 10,
    "CloudEnvironment": "AzureGovernment"
  }
}
```

## Feature 029 — Enterprise MCP Server Hardening

Adds 9 production-grade hardening capabilities to the MCP server.

### New Capabilities

#### 1. Resilient Retry Logic (US1)
- Polly 8.x retry + circuit breaker + timeout pipelines on all HTTP clients
- Configurable via `Resilience:Pipelines` in appsettings
- Automatic jitter on exponential backoff

#### 2. Rate Limiting (US2)
- Per-endpoint sliding-window rate limiting with per-client partitioning
- Returns `429 Too Many Requests` with `Retry-After` header
- Configurable via `RateLimiting:Policies`

#### 3. Path Sanitization (US3)
- `PathSanitizationService` validates all file path parameters
- Blocks path traversal (`..`), null bytes, and shell metacharacters
- `ScriptSanitizationService` for remediation script validation

#### 4. Response Caching (US4)
- `ResponseCacheService` with per-subscription scope and configurable TTL
- Cache hit/miss counters exposed via OpenTelemetry
- Stale-while-revalidate support

#### 5. Monitoring & Observability (US5)
- OpenTelemetry metrics: request duration, total requests, cache hits/misses
- Distributed tracing with `ActivitySource("Ato.Copilot.Mcp")`
- Optional Prometheus `/metrics` endpoint
- Enhanced `/health` with uptime, build version, and per-check timing

#### 6. Lazy Loading (US6)
- All 7 knowledge base services refactored to `Lazy<Task<T>>`
- Thread-safe initialization with `LazyThreadSafetyMode.ExecutionAndPublication`
- `IAsyncEnumerable<T>` streaming for compliance findings and SSP sections

#### 7. Pagination (US7)
- Server-side enforcement: default 50, max 100 items per page
- `PaginationInfo` metadata envelope with `nextPageToken` for cursor-based navigation
- `/mcp/tools` endpoint supports `page` and `pageSize` query parameters
- Automatic detection and mapping of `PagedResult<T>` from kanban/audit tools

#### 8. Offline Mode (US8)
- `OfflineModeService` for IL6 air-gapped operation
- NIST control lookups, STIG data, and cached assessments work offline
- AI-dependent operations return `OFFLINE_UNAVAILABLE` error code
- `/health` reports `Degraded` with capability lists when offline
- `CacheRepository` for persistent offline cache (EF Core)

#### 9. SSE Reconnection (US9)
- Monotonic event IDs on all SSE events
- `Last-Event-ID` header support for reconnection replay
- `SseEventBuffer` with 256-event per-session capacity
- Keepalive comments every 15 seconds to prevent proxy timeouts
- Automatic buffer eviction after session completion or 60s inactivity

## Breaking Changes

- **Paginated responses**: Previously unbounded collection responses are now limited to 50 items by default. Clients should check `metadata.pagination.hasNextPage` and use `nextPageToken` to retrieve additional pages.
- **SSE event format**: SSE events now include `id:` lines. Clients should store the last event ID for reconnection support.

## Configuration

New sections in `appsettings.json`:

| Section | Default | Description |
|---------|---------|-------------|
| `Resilience` | 3 retries, 2s delay | Retry/circuit breaker settings |
| `RateLimiting` | 30/60s | Rate limit policies |
| `Caching` | 50MB, 900s TTL | Response cache settings |
| `Pagination` | 50/100 | Page size defaults |
| `Streaming` | 256 buffer, 15s keepalive | SSE settings |
| `OpenTelemetry` | OTLP disabled | Metrics/tracing export |
| `Server.OfflineMode` | false | Offline mode toggle |
