## Feature 021: Privacy & Interconnection Management

ATO Copilot introduces **structured PTA/PIA management** for privacy threshold and impact analysis, **system interconnection registration** with ISA/MOU lifecycle tracking, and a **privacy compliance dashboard** that aggregates gate status for authorization readiness.

---

### New MCP Tools

| Tool | Description | RBAC |
|------|-------------|------|
| `compliance_create_pta` | Create a Privacy Threshold Analysis with PII categories, collection justification, and sharing details | ISSM, ISSO |
| `compliance_generate_pia` | Generate a Privacy Impact Assessment with 9 sections from PTA data, supporting iterative refinement | ISSO, ISSM |
| `compliance_review_pia` | Review and approve/reject a PIA with reviewer comments and audit trail | ISSM |
| `compliance_check_privacy_compliance` | Dashboard aggregating PTA, PIA, and interconnection agreement gate status | All compliance roles |
| `compliance_add_interconnection` | Register a system-to-system interconnection crossing the authorization boundary | ISSM, ISSO, Engineer |
| `compliance_list_interconnections` | List all interconnections for a system with status and agreement info | All compliance roles |
| `compliance_update_interconnection` | Update interconnection details, status, or data flow description | ISSM, ISSO |
| `compliance_generate_isa` | Generate an Interconnection Security Agreement document from interconnection data | ISSM |
| `compliance_register_agreement` | Register an ISA/MOU/MOA with document metadata and expiration tracking | ISSM |
| `compliance_update_agreement` | Update agreement status, expiration date, or renewal information | ISSM |
| `compliance_certify_no_interconnections` | Certify that a system has no external interconnections (sets HasNoExternalInterconnections flag) | ISSM |
| `compliance_validate_agreements` | Validate all interconnection agreements are current and not expired | ISSM, ISSO, SCA |

---

### Key Capabilities

#### Privacy Threshold Analysis (PTA)
- Structured PII category capture with justification and legal authority
- Collection method documentation (direct, derived, third-party)
- Sharing and retention period tracking
- Automatic PIA-required determination based on PII categories

#### Privacy Impact Assessment (PIA)
- 9-section PIA generation from PTA data following NIST guidance
- Lifecycle management: Draft → UnderReview → Approved → Expired
- Reviewer comments and approval workflow with audit trail
- Annual review tracking with expiration alerts

#### Interconnection Registration
- System-to-system connection documentation with direction (Inbound, Outbound, Bidirectional)
- Data flow description and security requirements capture
- Connection status lifecycle: Proposed → Active → Suspended → Terminated
- HasNoExternalInterconnections certification for systems with no boundary crossings

#### Agreement Lifecycle (ISA/MOU/MOA)
- Agreement document registration with type classification (ISA, MOU, MOA)
- Expiration date tracking with validation alerts
- Renewal workflow with document versioning
- Cross-system agreement linking

## Feature 022: SSP 800-18 Full Sections + OSCAL Output

ATO Copilot now generates a **complete 13-section System Security Plan** per NIST SP 800-18 Rev 1, adds structured SSP section management with per-section authoring lifecycle (Draft → UnderReview → Approved), and exports **OSCAL 1.1.2-compliant SSP JSON** with structural validation for FedRAMP and eMASS submission.

---

### New MCP Tools

| Tool | Description | RBAC |
|------|-------------|------|
| `compliance_write_ssp_section` | Create or update individual SSP sections with lifecycle tracking, optimistic concurrency, and optional submit-for-review workflow | ISSM, ISSO |
| `compliance_review_ssp_section` | Approve or request revision on SSP sections under review, with reviewer comments and audit trail | ISSM, ISSO |
| `compliance_ssp_completeness` | Check SSP readiness with per-section status, overall completion percentage, and blocking issues list | All compliance roles |
| `compliance_export_oscal_ssp` | Export a complete OSCAL 1.1.2 SSP JSON document with metadata, system-characteristics, system-implementation, control-implementation, and back-matter | All compliance roles |
| `compliance_validate_oscal_ssp` | Validate OSCAL SSP JSON against 7 structural checks with errors, warnings, and statistics | All compliance roles |

### Enhanced MCP Tools

| Tool | Changes |
|------|---------|
| `compliance_generate_ssp` | Now generates all 13 NIST 800-18 sections (up from 5), supports 13 new section keys + 3 backward-compatible keys, includes YAML front-matter metadata and completeness warnings |
| `compliance_export_oscal` | SSP export now delegates to `OscalSspExportService` for full OSCAL 1.1.2 compliance; Assessment Results and POA&M exports unchanged |

---

### Key Capabilities

#### SSP Section Management (Cap 1.1–1.4)
- Per-section lifecycle: NotStarted → Draft → UnderReview → Approved
- Optimistic concurrency with version tracking — rejects conflicting writes
- Auto-generated sections (§1, §2, §3, §4, §7, §9, §10, §11) pull content from existing entities
- Authored sections (§5, §8, §12, §13) support free-form markdown with metadata
- Hybrid sections (§6) merge auto-populated environment data with authored narrative
- Manual override support for auto-generated sections with `HasManualOverride` flag
- Completeness tracking with readiness percentage and blocking issues

#### Complete SSP Generation (Cap 2.1–2.10)
- **§1 System Identification** — DitprId, EmassId, system name, key personnel table
- **§2 Security Categorization** — FIPS 199 levels with FIPS 200 minimum requirements reference
- **§3 System Owner / Authorizing Official** — Detailed personnel from RmfRoleAssignment with responsibilities
- **§4 Information System Type** — Major Application vs GSS, mission criticality, operational status with dates
- **§5 General Description / Purpose** — Authored section with template fallback from RegisteredSystem.Description
- **§6 System Environment** — Hybrid: auto-populated cloud region + authored physical/logical environment detail
- **§7 System Interconnections** — Repositioned Feature 021 data with connection table and sharing narrative
- **§8 Related Laws & Regulations** — Authored section for applicable legal/regulatory requirements
- **§9 Minimum Security Controls** — Baseline level, overlay, control tailoring rationale, compensating controls
- **§10 Control Implementations** — Full control narrative detail (existing capability)
- **§11 Authorization Boundary** — Resource inventory from AuthorizationBoundary entities + authored narrative
- **§12 Personnel Security** — Auto-populated roles + authored screening, training, and separation procedures
- **§13 Contingency Planning** — ContingencyPlanReference metadata (RTO, RPO, test date) + authored narrative

#### OSCAL 1.1.2 SSP Export (Cap 3.1–3.8)
- Complete OSCAL SSP JSON with all 6 required top-level sections:
  - `metadata` — title, oscal-version "1.1.2", roles, parties, responsible-parties
  - `import-profile` — static baseline URIs (Low/Moderate/High) per NIST catalog
  - `system-characteristics` — security levels, information types, boundary, network architecture
  - `system-implementation` — components from AuthorizationBoundary, inventory items, users, leveraged authorizations
  - `control-implementation` — implemented requirements with by-components cross-references
  - `back-matter` — ISA/MOU documents, PIA, contingency plan, SSP self-reference
- Partial export with schema-valid placeholders when data is incomplete
- Warning collection for missing data gaps without blocking export
- Statistics reporting (control, component, inventory item, user, back-matter resource counts)

#### OSCAL Structural Validation (Cap 4.1–4.3)
- 7 structural validation checks:
  1. Top-level `system-security-plan` key present
  2. Required child sections present (metadata, import-profile, system-characteristics, system-implementation, control-implementation)
  3. All UUID values are valid GUID format
  4. Control-ids match import-profile baseline controls
  5. Component UUIDs in by-components resolve in system-implementation
  6. Party UUIDs in responsible-parties resolve in metadata
  7. OSCAL version equals "1.1.2"
- Errors (structural failures) vs Warnings (non-blocking issues) separation

#### Enhanced SSP Generation (Cap 5.1–5.5)
- 13 new section keys with backward-compatible mapping (e.g., `baseline` → `minimum_controls`)
- YAML front-matter metadata block: document version, generation timestamp, categorization level, section completion
- Completeness markers: `[NOT STARTED]`, `[UNAPPROVED]`, `[MANUAL OVERRIDE]`
- Selective section generation via `sections` parameter
