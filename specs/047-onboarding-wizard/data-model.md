# Data Model: Tenant Onboarding Wizard (Feature 047)

**Date**: 2026-05-07
**Spec**: [spec.md](./spec.md) · **Plan**: [plan.md](./plan.md) · **Research**: [research.md](./research.md)

## Conventions

- Every entity is defined inside `src/Ato.Copilot.Core/Models/Onboarding/` and added as a new
  `DbSet<T>` on `AtoCopilotContext`.
- `Id` columns are `Guid` (`uniqueidentifier` / `TEXT` depending on provider) generated by the
  application unless noted.
- All entities are **tenant-scoped**: every row carries `TenantId : Guid` (FK to existing tenant
  registry) and is indexed by it. Authorization filters add a `WHERE TenantId = @currentTenantId`
  predicate at query time.
- `CreatedAt` / `CreatedBy` / `UpdatedAt` / `UpdatedBy` are present on every entity that admin
  actions can mutate. They use `DateTimeOffset` and the actor's user id (`Guid` matching the
  `Person.EntraObjectId` when directory-linked, or the `Person.Id` otherwise).
- All file-storage references (`StorageBlobKey`) are opaque strings whose layout is defined in
  `plan.md` § Storage container layout. Bytes live in `IFileStorageProvider`; metadata lives
  here.
- "Soft delete" means `DeletedAt : DateTimeOffset?` is set and the row is excluded by global
  query filter; "hard delete" deletes the row.

### 1. `TenantOnboardingState`

One row per tenant. Drives whether the wizard auto-opens, where it resumes, and whether re-runs
are recorded.

| Column | Type | Notes |
|--------|------|-------|
| `Id` | `Guid` (PK) |  |
| `TenantId` | `Guid` (UQ) | One row per tenant. |
| `Status` | `enum` | `NotStarted` / `InProgress` / `Completed` / `ReRunInProgress`. |
| `CompletedSteps` | `string` (CSV) or 1:N child table | Set of completed step ids. Implementation detail: stored as `OnboardingStepCompletion` rows for indexability. |
| `LastStep` | `string` | Last step user was on; powers resume. |
| `OnboardingStartedAt` | `DateTimeOffset?` |  |
| `OnboardingCompletedAt` | `DateTimeOffset?` | Set when all required steps complete. |
| `LastReRunAt` | `DateTimeOffset?` |  |
| `CreatedAt` / `CreatedBy` / `UpdatedAt` / `UpdatedBy` | (standard) |  |

**Validation rules**: `Status = Completed` ⇒ `OnboardingCompletedAt != null`. Required steps

**State transitions**:

```text
NotStarted ─► InProgress ─► Completed ─► ReRunInProgress ─► Completed
                ▲                                              │
                └──────────────────────────────────────────────┘  (admin re-runs allowed at any time)
```

## Default-template "exactly one" invariant

Service-level transaction on default toggle (`OrganizationTemplateService.SetDefaultAsync`):

1. `BEGIN TX`
2. `UPDATE OrganizationDocumentTemplates SET IsDefault = 0 WHERE TenantId = @t AND TemplateType = @type AND Id <> @id`
3. `UPDATE OrganizationDocumentTemplates SET IsDefault = 1 WHERE Id = @id`
4. Append `WizardAuditEntry { Action = TemplateMarkedDefault, EffectsJson = "...prevDefaultId..." }`
5. `COMMIT`

This pairs with the filtered unique index for belt-and-braces enforcement.

## Cascade flagging algorithm (FR-094, R6)

```text
On replace(sourceArtifactId, newBlobKey, newChecksum, newVersionString):
  1. ValidateAdminPolicy()
  2. UPDATE source artifact metadata (StorageBlobKey, Checksum/Version, UpdatedAt/By)
     — DO NOT delete the previous blob until the admin confirms (FR-091 retention)
  3. SELECT * FROM WizardArtifactDependency
     WHERE TenantId = @t
       AND SourceArtifactType = @type
       AND SourceArtifactId   = @id
       AND IsStale            = 0
  4. For each dependency row:
       - SET IsStale = 1, StaleSince = NOW(), StaleReason = humanize(jobType, prevTag, newTag)
  5. INSERT WizardAuditEntry { Action = TemplateReplaced/EmassReplaced/..., EffectsJson = stale-set summary }
  6. RETURN cascade summary { dependentsFlagged: N, suggestedReRunJobs: [...] }
```

Re-run is admin-initiated via `POST /api/onboarding/dependencies/{id}/rerun` which enqueues a
`WizardJobStatus` of the appropriate `JobType` and surfaces SignalR progress (R2).
